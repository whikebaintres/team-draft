# 팀 드래프트 — 가챠 드래프트 & 관리자 점수 등록

e스포츠 동호회용 랜덤 팀 뽑기 시스템. `gacha.html`(드래프트), `admin.html`(관리자 점수 등록), `result.html`(결과 확인) 세 페이지가 Firebase Realtime Database 하나를 공유 데이터스토어로 써서 서로 연결되어 있다.

> 이 레포에는 이전에 만든 핀볼 방식 드래프트(`index.html` / `state.json`)도 함께 있지만, 이 문서는 이후에 추가된 가챠 드래프트 시스템(`gacha.html`, `admin.html`, `result.html`)만 다룬다. 두 시스템은 서로 다른 데이터스토어를 쓰기 때문에 충돌하지 않는다.

## 전체 흐름

1. **`gacha.html`** 에서 참가자 명단을 관리하고 팀 수를 정한 뒤 드래프트를 실행한다.
2. 카드가 한 명씩 등장하며 팀에 배정되고, 결과가 나오면 (년월/정식 모드일 경우) Firebase에 이번 드래프트 결과가 자동 공유된다. 토큰이나 로그인이 전혀 필요 없다. 종료 화면에는 선봉전~대장전 대진표가 뜨고, "결과 확인하기"로 바로 `result.html`로 이동할 수 있다.
3. **`admin.html`** 에서 그 결과를 조회하고, 실제 경기 후 각 라운드의 두 선수 점수를 입력한다. 등록을 누르면 `result.html`로 이동해 방금 등록한 결과를 바로 확인할 수 있다.
4. 등록된 점수는 Firebase의 누적 전적(`playerStats`, `headToHead`)에 반영되고, 다음에 `gacha.html`을 다시 열면 카드에 갱신된 수치가 보인다.
5. **`result.html`** 에서는 언제든 년월을 골라 그 달의 팀별 누적 점수/승점과 라운드별 결과를 조회할 수 있다.

## 페이지별 기능

### `gacha.html` — 드래프트 페이지

- **참가자 관리**: 추가/삭제 (기본 11명), 변경된 항목만 애니메이션 (전체 리스트 깜빡임 없음)
- **팀 구성**: 최소 2팀 ~ 최대 8팀, 팀 이름은 국내 e스포츠 구단명 중 랜덤 배정
- **기록 모드 스위치** (팀 구성 패널 상단): 이번 달 년월 라벨(예: "2026년 7월", 매달 자동 갱신) 또는 "테스트" 중 선택
  - **년월(정식) 모드**: 드래프트가 끝나면 결과가 Firebase에 공유되어 관리자 페이지에서 채점 가능
  - **테스트 모드**: 결과가 전혀 공유되지 않음 (Firebase에 아무 요청도 안 감) — 연습/테스트용
  - "새로 시작하기"를 누르면 항상 정식 모드로 자동 리셋됨 (테스트 모드가 실수로 이어지는 것 방지)
- **드래프트 연출** (철권 VS 화면 참고):
  - 왼쪽 선수 카드 등장 → **3초** 대기 → 상대(오른쪽) 카드 등장 → 약 0.9초 후 VS 마크 + 포지션 라벨 + 상대전적 + 해설 문구 동시 공개 → 5초 대기 → 다음 페어로 전환
  - 포지션 라벨: **선봉전 → 차봉전 → 중견전 → 부장전 → 대장전** (전체 5쌍일 때). 정확히 4쌍이면 중견전을 빼고 선봉전·차봉전·부장전·대장전만 사용
  - 카드: 라운딩 사각 카드(팀 컬러 전체 스트로크 + 글로우), 개인 승/패/승률/POM 표시. 포트레이트 자리에는 `players/` 하위의 선수별 실사 사진(회색 톤 보정 + 원형 마스크 처리된 PNG, `PLAYER_IMAGES` 맵으로 연결)이 뜨고, 사진이 없는 선수는 기존 실루엣 아이콘으로 폴백. 포트레이트는 얼굴이 원형 정중앙에 오도록 크롭됨. 현재 이도현·임준·김탁우·이예은·편한슬·고재형·원종훈·조이진·지한규·유다현·임별 11명 등록됨 (원본: `img/` 단체사진, 색깔 원 또는 머리 위 이름표 기준)
  - **상대전적**: 이번 매치업 두 사람의 과거 전적을 "OOO 2승 0패로 우위" 처럼 앞서는 사람 이름을 명시해서 표시, 첫 대결이면 "첫 맞대결"
  - **해설 문구**: 상대전적 상황(첫 대결/막상막하/압도적 우위)에 따라 다양한 문구가 랜덤하게 노출
- **BGM / 효과음**: "드래프트 시작하기" 클릭 시 지정된 유튜브 영상이 배경음악으로 재생 (숨김 플레이어). 우측 상단 스피커 아이콘 클릭으로 음소거, 호버 시 세로 볼륨 슬라이더 노출. 카드가 확정될 때마다 `FFXIV_Confirm.mp3`, 상대전적이 공개될 때 `charge.mp3` 효과음 재생. 이 두 효과음 볼륨은 고정값이 아니라 BGM 볼륨 슬라이더 값과 실시간으로 동기화됨 (슬라이더를 옮기면 즉시 같이 바뀜)
- **공유**: 별도 설정 없이 항상 자동 공유됨 (년월/정식 모드일 때). 토큰이나 로그인 불필요
- **종료 화면**: 팀 로스터 카드 대신, 선봉전~대장전 라운드별 VS 매치업 리스트로 대진표를 보여줌. 정식 모드로 공유가 끝나면 "새로 시작하기" 옆에 "결과 확인하기" 버튼이 나타나 `result.html`로 바로 이동할 수 있음 (테스트 모드는 결과가 공유되지 않으므로 이 버튼이 뜨지 않음)

### `admin.html` — 관리자 점수 등록 페이지

- **년월 선택**: 지금까지 등록된 드래프트 중 년월을 드롭다운으로 골라 조회/수정할 수 있음 (최근 등록 순 정렬). 초기 시드로 넣은 과거 파싱 데이터(`asset/DATABASE.xlsx` 기반)는 날짜 정보가 없어 드롭다운에 나오지 않음
- **비밀번호 게이트**: 점수를 등록/수정/삭제하려면 비밀번호(`daft`)를 입력해야 함. 조회는 비밀번호 없이 누구나 가능하지만, 점수 입력 칸·등록 버튼·삭제 버튼은 잠금 해제 전엔 비활성화됨. Firebase 자체는 여전히 완전 오픈이라 이건 실수 방지용 게이트일 뿐, 진짜 보안은 아님. 탭 세션 동안(`sessionStorage`)은 다시 물어보지 않음
- **라운드별 단일 표**: 팀별 두 단락으로 나뉘어 있던 기존 방식 대신, 대진(`pairs`) 기준 라운드별(선봉전~대장전)로 두 대결자를 한 카드에 묶어서 보여줌
- **점수 입력 → 승/패/무 자동 판정**: 각 대결자에게 숫자 점수 입력칸(십만 단위 이상도 들어가도록 넉넉한 너비)을 주고, 양쪽 점수가 모두 입력되면 비교해서 승/패/무를 자동으로 결정 (라디오 버튼으로 직접 승/패/무를 고르지 않음). 한쪽만 입력된 라운드는 반영되지 않음 (부분 입력 허용)
- **POM 없음**: POM 선정은 이 페이지에서 다루지 않음 (별도 페이지에서 관리 예정). 이 페이지는 승/패/무 raw 데이터만 등록하며, 과거에 기록된 POM 플래그가 있다면 건드리지 않고 그대로 보존함
- 등록하면 `playerStats`(개인 누적 승/패/무)와 `headToHead`(매치업 쌍별 상대전적)가 함께 갱신되고, `result.html`의 해당 년월 화면으로 바로 이동한다
- 이미 채점된 결과는 읽기 전용으로 보여주고 "다시 채점하기"로 재오픈 가능 (재채점 시 이전 기여분을 먼저 되돌린 뒤 새로 반영)
- **이 달 데이터 삭제**: 점수 등록 버튼 옆의 삭제 버튼(확인창 포함)으로 선택된 년월의 드래프트를 통째로 지울 수 있음. 이미 채점된 상태였다면 `playerStats`/`headToHead`에 반영된 기여분을 먼저 되돌린 뒤 `drafts`에서 항목을 제거한다 (되돌릴 수 없는 작업)

### `result.html` — 결과 확인 페이지

- **년월 선택**: admin.html과 동일한 드롭다운으로 년월을 골라 조회하며, 기본값은 가장 최근에 등록된 년월이다. `admin.html`에서 등록 직후 넘어올 때는 `?ym=년월` 쿼리로 방금 등록한 달을 최초 1회 우선 선택한다
- **팀 스코어보드**: 팀별로 "누적 점수"(그 팀 소속 선수들의 라운드 점수 합계)와 "승점"(선봉전~대장전 중 그 팀 소속 선수가 이긴 라운드 수, 1승당 1점)을 함께 보여준다
- **라운드별 결과 요약**: 라운드별로 "고재형 3 : 1 유다현" 한 줄씩 간단한 표로 나열 (이긴 쪽은 강조, 무승부는 별도 색)
- 비밀번호 없이 누구나 조회만 가능한 페이지이며 점수 수정 기능은 없다
- **최종 결과 확인 (POM 추첨)**: 아직 확정되지 않은 달이면 버튼을 눌렀을 때 모달이 뜨고, "누적 점수"/"승점" 중 하나를 5초간 슬롯처럼 굴리다 무작위로 하나를 확정한다. 확정되면 양쪽에서 폭죽 효과 + `FFXIV_Level_Up.mp3` 효과음(50% 볼륨)과 함께, 그 기준으로 가장 높은 팀과 팀 명단이 뜨고, 그 팀에서 개인 누적 점수가 가장 높은 사람이 POM으로 화려하게 강조된다. 이 결과는 `drafts`의 `finalResult`와 `history`에 저장되며(`playerStats`의 POM 카운트도 함께 증가), 한 번 확정된 달은 다시 열어도 재추첨·재연출 없이 저장된 결과만 보여준다. 여러 사람이 동시에 열어도 먼저 저장된 결과가 이기고 나머지는 그 결과를 그대로 따른다
  - `FFXIV_Level_Up.mp3`는 `charge.mp3`/`FFXIV_Confirm.mp3`와 같은 레포 루트에 있다 (원본은 `asset/FFXIV_Level_Up.mp3`)

## 데이터 모델 — Firebase Realtime Database (`/gachaState`)

```json
{
  "playerStats": {
    "고재형": { "win": 12, "loss": 6, "draw": 0, "pom": 2 }
  },
  "headToHead": {
    "고재형|지한규": { "a": "고재형", "b": "지한규", "aWin": 2, "bWin": 0, "draw": 0 }
  },
  "drafts": {
    "2026년 7월": {
      "runId": "d...",
      "createdAt": 1730000000000,
      "yearMonth": "2026년 7월",
      "scored": false,
      "teams": [{ "name": "T1", "color": "#0a5cd8", "colorName": "딥 블루", "members": ["..."] }],
      "pairs": [["이름A", "이름B"], ["이름C", "이름D"]],
      "results": null
    }
  },
  "latestDraft": { "...": "drafts의 가장 최근 항목과 동일한 값 (구버전 호환용, 아래 설명 참고)" },
  "history": {
    "2026년 7월": {
      "runId": "d...", "yearMonth": "2026년 7월", "createdAt": 1730000000000,
      "teams": ["..."], "pairs": ["..."], "results": ["..."],
      "finalResult": { "criteria": "totalScore", "teamName": "T1", "pomName": "이름A", "decidedAt": 1730000500000 },
      "archivedAt": 1730000500000
    }
  }
}
```

- `playerStats`: 참가자별 누적 전적. 최초 시드는 `asset/DATABASE.xlsx`(과거 A팀/B팀 대결 기록, POM은 쉼표로 구분된 이름 목록)를 파싱해서 만들었다. 이 시드 데이터는 `drafts`에 대응하는 항목이 없어(=날짜 정보가 없어) admin.html의 년월 드롭다운에는 나타나지 않는다.
- `headToHead`: 두 사람이 실제로 맞붙어서 나온 승/패/무 누적. 키는 두 이름을 `|`로 이은 것이며 순서는 상관없이 양방향으로 조회 가능하게 설계되어 있다.
- `drafts`: 년월(`yearMonth`)을 키로 하는 드래프트 히스토리. `gacha.html`에서 새 드래프트를 돌릴 때마다 해당 년월 키에 기록되며, admin.html은 이 맵 전체를 드롭다운으로 보여주고 어떤 월이든 선택해 채점/재채점할 수 있다. 같은 달에 두 번 드래프트를 돌리면 그 달의 항목은 덮어써진다 (달 단위로는 한 건만 보관). admin.html에서 "이 달 데이터 삭제"를 누르면 이 맵에서 항목이 통째로 사라진다.
  - `results`: 라운드(=`pairs`의 각 쌍)별로 두 선수의 점수를 비교해 자동으로 산출된 `{ 이름: { score, outcome } }` 형태. `outcome`은 `"win" | "loss" | "draw"`. `result.html`에서 POM이 확정되면 그 사람의 항목에 `pom: true`가 추가된다.
  - `finalResult`: `result.html`에서 "최종 결과 확인" 슬롯이 멈추고 나면 기록되는 `{ criteria, teamName, pomName, decidedAt }`. `criteria`는 `"totalScore" | "winPoints"` 중 슬롯이 뽑은 기준이고, `teamName`/`pomName`은 그 기준으로 정해진 우승팀/POM이다. 한 번 정해지면 고정이며, `result.html`은 이 필드가 있으면 다시 뽑지 않고 저장된 값을 그대로 보여준다.
- `latestDraft`: 이 기능(년월별 히스토리, 2026-07 도입) 이전에 쓰이던 필드로, `drafts`와 동일한 값을 가리키는 구버전 호환용 미러다. `gacha.html`은 새 드래프트마다 `drafts`와 `latestDraft`를 함께 갱신하고, admin.html은 `drafts`에 아직 없는 년월이 `latestDraft`에 있으면(=이 기능 도입 전에 생성된 데이터) 폴백으로 병합해 보여준다. 새로 저장되는 채점 결과는 `drafts`에만 반영된다.
- `history`: `result.html`에서 POM까지 확정된(=`finalResult`가 생긴) 드래프트만 영구 보관하는 별도 아카이브. `drafts`와 달리 admin.html의 "이 달 데이터 삭제"로도 지워지지 않는다 — 채점이 끝난 달의 최종 기록을 나중에 그 달 데이터를 정리하더라도 잃지 않기 위한 것. 년월당 한 번만 기록되며(멱등), 아직 없는 경우에만 채워진다.
- `pairs`는 각 대결이 실제로 누구와 누구였는지를 기록해, 관리자 점수 등록 시 `headToHead`를 자동으로 갱신하는 데 쓰인다.

## 공유 방식 (Firebase Realtime Database, 완전 오픈)

두 페이지 모두 별도 백엔드 없이, Firebase Realtime Database REST API로 직접 읽고/쓴다.

- URL: `https://team-draft-1ab3a-default-rtdb.firebaseio.com/gachaState.json`
- 규칙: `{ ".read": true, ".write": true }` — 인증/토큰 없이 누구나 읽고 쓸 수 있다. 친구 그룹 캐주얼 트래커라 보안보다 편의성을 우선한 선택.
- 읽기: `fetch(FIREBASE_URL + "/gachaState.json")`, 쓰기: 같은 경로에 `PUT` (전체 객체를 통째로 덮어씀).
- 예전에는 이 레포의 `gacha-state.json`을 GitHub Contents API로 읽고/쓰며 GitHub 토큰이 필요했으나, 지금은 Firebase로 완전히 이전해 토큰 개념이 없다. (핀볼 페이지 `index.html`/`state.json`은 여전히 예전 GitHub 방식을 그대로 쓴다.)
- 레포에 남아있는 `gacha-state.json`은 더 이상 앱이 읽지 않는 **과거 시드 데이터 스냅샷**일 뿐이다 (참고용).

## 알려진 제한 사항

- `drafts`는 같은 달에 새 드래프트를 다시 돌리면 그 달 항목이 덮어써진다 (달 단위 히스토리이지, 달 안에서 여러 건을 보관하진 않는다).
- 팀 수가 2보다 크면 화면상의 "왼쪽 vs 오른쪽" 페어링이 실제 팀 간 대결과 정확히 일치하지 않을 수 있다 (연출/연속 등장 순서 기준이라 2팀일 때 가장 자연스럽다).
- **완전 오픈 데이터스토어이므로 URL을 아는 사람은 누구나 데이터를 읽고 쓸 수 있다.** 토큰 없이 편하게 쓰는 대신 감수한 트레이드오프. admin.html의 점수 수정 비밀번호(`daft`)는 클라이언트 측 실수 방지 게이트일 뿐, Firebase 규칙 자체를 막지는 못한다 — 비밀번호를 몰라도 REST API로 직접 쓰면 우회 가능하다.
- 이 README가 다루는 기능은 로컬 정적 서버 또는 GitHub Pages 배포본에서 검증되었다. 로컬 파일 더블클릭(`file://`)으로는 Firebase fetch가 막혀 전적 연동이 동작하지 않는다.
