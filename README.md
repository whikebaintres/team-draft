# 팀 드래프트 — 가챠 드래프트 & 관리자 페이지

e스포츠 동호회용 랜덤 팀 뽑기 시스템. 사용자용 페이지 `gacha.html`(드래프트)·`result.html`(결과 확인)과 관리자용 페이지 `admin-home.html`(대시보드)·`admin.html`(점수 등록)·`admin-members.html`(회원 관리)·`admin-stats.html`(회원 데이터)가 Firebase Realtime Database 하나를 공유 데이터스토어로 써서 서로 연결되어 있다.

> 이 레포에는 이전에 만든 핀볼 방식 드래프트(`index.html` / `state.json`)도 함께 있지만, 이 문서는 이후에 추가된 가챠 드래프트 시스템만 다룬다. 두 시스템은 서로 다른 데이터스토어를 쓰기 때문에 충돌하지 않는다.

## 전체 흐름

1. **`admin-members.html`** 에서 회원을 이름·사진과 함께 등록해둔다. 여기 등록된 회원이 곧 드래프트의 기본 참가자 명단이 된다.
2. **`gacha.html`** 에서 그 명단을 확인하고(필요하면 그 자리에서 추가/삭제) 드래프트를 실행한다.
3. 카드가 한 명씩 등장하며 두 팀에 배정되고, 결과가 나오면 (년월/정식 모드일 경우) Firebase에 이번 드래프트 결과가 자동 공유된다. 토큰이나 로그인이 전혀 필요 없다. 종료 화면에는 선봉전~대장전 대진표가 뜨고, "결과 확인하기"로 바로 `result.html`로 이동할 수 있다.
4. **`admin.html`** 에서 그 결과를 조회하고, 실제 경기 후 각 라운드의 두 선수 점수를 입력한다. 등록을 누르면 `result.html`로 이동해 방금 등록한 결과를 바로 확인할 수 있다.
5. 등록된 점수는 Firebase의 누적 전적(`playerStats`, `headToHead`)에 반영되고, 다음에 `gacha.html`을 다시 열면 카드에 갱신된 수치가 보인다. 누적 전적은 **`admin-stats.html`** 에서 조회하고 필요하면 직접 고칠 수 있다.
6. **`result.html`** 에서는 언제든 년월을 골라 그 달의 팀별 누적 점수/승점과 라운드별 결과를 조회할 수 있다.

관리자 페이지 세 개(`admin.html`, `admin-members.html`, `admin-stats.html`)는 **`admin-home.html`** 대시보드에서 한 번에 진입할 수 있다.

## 페이지별 기능

### `gacha.html` — 드래프트 페이지

- **참가자 관리**: 기본 명단은 `admin-members.html`에서 등록한 회원(숨김 처리되지 않은 사람) 목록을 Firebase `/members`에서 읽어와 채운다. 불러오기 전/실패 시에는 페이지에 하드코딩된 11명으로 폴백한다. 이 화면에서 그때그때 추가/삭제한 것은 이번 판에만 적용되며 회원 목록에는 반영되지 않는다 (한 번이라도 손대면 뒤늦게 도착한 회원 목록이 그 명단을 덮어쓰지 않는다). 변경된 항목만 애니메이션 (전체 리스트 깜빡임 없음)
- **팀 구성**: 항상 2팀 고정 (팀 수 선택 UI 없음). 팀 이름은 국내 e스포츠 구단명 중 랜덤 배정
- **기록 모드 스위치** (참가자 패널 상단): 이번 달 년월 라벨(예: "2026년 7월", 매달 자동 갱신) 또는 "테스트" 중 선택
  - **년월(정식) 모드**: 드래프트가 끝나면 결과가 Firebase에 공유되어 관리자 페이지에서 채점 가능
  - **테스트 모드**: 결과가 전혀 공유되지 않음 (Firebase에 아무 요청도 안 감) — 연습/테스트용
  - "새로 시작하기"를 누르면 항상 정식 모드로 자동 리셋됨 (테스트 모드가 실수로 이어지는 것 방지)
- **드래프트 연출** (철권 VS 화면 참고):
  - 왼쪽 선수 카드 등장 → **3초** 대기 → 상대(오른쪽) 카드 등장 → 약 0.9초 후 VS 마크 + 포지션 라벨 + 상대전적 + 해설 문구 동시 공개 → 5초 대기 → 다음 페어로 전환
  - 포지션 라벨: **선봉전 → 차봉전 → 중견전 → 부장전 → 대장전** (전체 5쌍일 때). 정확히 4쌍이면 중견전을 빼고 선봉전·차봉전·부장전·대장전만 사용. 라벨은 카드 위쪽에 여백을 두고 배치돼 카드와 겹치지 않음
  - 카드: 라운딩 사각 카드(팀 컬러 전체 스트로크 + 글로우), 개인 승/패/승률/POM 표시. 포트레이트는 회원 목록의 `photo` 값으로 채워지며(레포 안 파일 경로이거나, 업로드된 사진의 data URL), 사진이 없는 선수는 실루엣 아이콘으로 폴백. 마스킹은 CSS `border-radius: 50%`로 처리하므로 원본 이미지가 원형일 필요는 없다. 초기 11명(이도현·임준·김탁우·이예은·편한슬·고재형·원종훈·조이진·지한규·유다현·임별)은 `players/` 하위 PNG 경로를 그대로 참조한다 (원본: `img/` 단체사진 기준. `players/new/`는 원본 스테이징 폴더로 `.gitignore` 처리되어 커밋 제외)
  - **상대전적**: 이번 매치업 두 사람의 과거 전적을 항상 "상대 전적 ·" 접두어와 함께 표시 (예: "상대 전적 · OOO 2승 0패로 우위"). 개인 통산 기록과 혼동되지 않게 명시. 첫 대결이면 "상대 전적 · 첫 맞대결"
  - **해설 문구**: 상대전적 상황(첫 대결/막상막하/압도적 우위)에 따라 다양한 문구가 랜덤하게 노출
- **BGM / 효과음**: "드래프트 시작하기" 클릭 시 지정된 유튜브 영상이 배경음악으로 재생 (숨김 플레이어). 우측 상단 스피커 아이콘 클릭으로 음소거, 호버 시 세로 볼륨 슬라이더 노출. 카드가 확정될 때마다 `FFXIV_Confirm.mp3`, 상대전적이 공개될 때 `charge.mp3` 효과음 재생. 두 효과음 볼륨은 BGM 볼륨 슬라이더 값과 실시간 동기화되며, `charge.mp3`(상대전적 공개음)는 상대적으로 커서 카드 확정음의 **절반(50%)** 크기로 재생됨
- **공유**: 별도 설정 없이 항상 자동 공유됨 (년월/정식 모드일 때). 토큰이나 로그인 불필요
- **종료 화면**: 팀 로스터 카드 대신, 선봉전~대장전 라운드별 VS 매치업 리스트로 대진표를 보여줌. 한눈에 들어오도록 라운드당 한 줄(`[이름 팀] VS [이름 팀]`)로 압축된 컴팩트 레이아웃. 정식 모드로 공유가 끝나면 "새로 시작하기" 옆에 "결과 확인하기" 버튼이 나타나 `result.html`로 바로 이동할 수 있음 (테스트 모드는 결과가 공유되지 않으므로 이 버튼이 뜨지 않음)

### `admin-home.html` — 관리자 대시보드

- 관리자 페이지 세 개(점수 등록 / 회원 관리 / 회원 데이터)로 가는 진입 카드와, 현재 데이터 요약(등록 회원 수, 드래프트 건수, 채점 완료 건수, 최근 년월)을 보여준다
- **비밀번호 게이트**: 여기서 한 번 잠금을 풀면 `sessionStorage`의 같은 키(`adminUnlocked`)를 공유하는 세 관리자 페이지가 모두 풀린 상태가 된다. "다시 잠그기"로 세션 잠금을 되돌릴 수 있다

### `admin-members.html` — 회원 관리 페이지

- **신규 등록**: 이름(최대 12자)과 사진을 넣어 회원을 등록한다. 등록 즉시 `gacha.html`의 기본 참가자 명단에 나타난다
- **사진 업로드**: 파일 선택 또는 드래그앤드롭. 원본을 캔버스로 **중앙 정사각 크롭 → 320×320 → JPEG(품질 0.85)** 로 줄여 data URL로 저장한다 (1.6MB 원본 PNG 기준 약 15KB). gacha 카드가 CSS로 원형 마스킹하므로 투명도가 필요 없어 JPEG를 쓴다. 사진 없이 등록하면 실루엣 아이콘으로 표시된다
- **목록/수정**: 등록된 회원을 썸네일·이름·전적 요약과 함께 보여주고, "수정"으로 이름 변경 / 사진 교체·삭제 / 숨김 전환을 할 수 있다
  - **숨김**: 기록은 그대로 두고 드래프트 기본 명단에서만 빼는 상태. 잠시 활동을 쉬는 회원에 쓴다
  - **이름 변경**: `gachaState` 전체(`playerStats` 키, `headToHead` 키와 `a`/`b` 필드, `drafts`/`history`/`latestDraft`의 `teams[].members`·`pairs`·`results` 키·`finalResult.pomName`)를 한 번에 치환한다. 이름을 바꿔도 지난 전적이 끊기지 않는다. 치환 결과 이미 존재하는 상대 쌍과 키가 겹치면 두 기록을 합산한다
  - **삭제**: 회원 목록에서만 제거하고 누적 전적·드래프트 기록은 남긴다 (기록까지 지우려면 `admin-stats.html`에서 값을 0으로 만든다)
- **기존 11명 자동 이관**: `/members`가 비어 있으면 최초 1회, gacha.html에 하드코딩돼 있던 11명을 `players/*.png` 경로와 함께 자동으로 등록한다

### `admin-stats.html` — 회원 데이터 페이지

- **누적 전적 표**: 회원별 경기 수 / 승 / 패 / 무 / 승률 / POM을 한 표로 보여준다. 경기 수·승수·승률·POM·이름순으로 정렬 가능
- **상대전적 펼치기**: 이름을 누르면 그 회원의 `headToHead` 기록이 상대별로 펼쳐진다 (맞대결이 많은 순)
- **값 직접 수정**: 잠금 해제 후 "값 수정하기"를 누르면 승/패/무/POM이 입력칸으로 바뀌고, 값을 고치면 경기 수·승률이 즉시 다시 계산된다. 저장은 **바뀐 사람만 골라 `PATCH`** 하므로 그 사이 다른 사람이 등록한 채점 결과를 덮어쓰지 않는다
- 전체 요약(기록 있는 인원 / 누적 대결 / 무승부 / POM 합계)도 함께 보여준다. 한 경기에 두 명이 기록되므로 대결·무승부 수는 합계의 절반으로 환산한다
- 회원 목록에 없는 이름(초기 시드처럼 `playerStats`에만 있는 경우)도 "회원 미등록" 태그와 함께 표시된다
- 여기서 고치는 값은 누적 전적을 직접 덮어쓰는 것이라 라운드별 기록(`drafts[*].results`)과는 어긋날 수 있다. 라운드 단위 수정은 `admin.html`에서 한다

### `admin.html` — 관리자 점수 등록 페이지

- **년월 선택**: 지금까지 등록된 드래프트 중 년월을 드롭다운으로 골라 조회/수정할 수 있음 (최근 등록 순 정렬). 초기 시드로 넣은 과거 파싱 데이터(`asset/DATABASE.xlsx` 기반)는 날짜 정보가 없어 드롭다운에 나오지 않음
- **비밀번호 게이트**: 점수를 등록/수정/삭제하려면 비밀번호(`daft`)를 입력해야 함. 조회는 비밀번호 없이 누구나 가능하지만, 점수 입력 칸·등록 버튼·삭제 버튼은 잠금 해제 전엔 비활성화됨. Firebase 자체는 여전히 완전 오픈이라 이건 실수 방지용 게이트일 뿐, 진짜 보안은 아님. 탭 세션 동안(`sessionStorage`)은 다시 물어보지 않음
- **라운드별 단일 표**: 팀별 두 단락으로 나뉘어 있던 기존 방식 대신, 대진(`pairs`) 기준 라운드별(선봉전~대장전)로 두 대결자를 한 카드에 묶어서 보여줌
- **점수 입력 → 승/패/무 자동 판정**: 각 대결자에게 숫자 점수 입력칸(십만 단위 이상도 들어가도록 넉넉한 너비)을 주고, 양쪽 점수가 모두 입력되면 비교해서 승/패/무를 자동으로 결정 (라디오 버튼으로 직접 승/패/무를 고르지 않음). 한쪽만 입력된 라운드는 반영되지 않음 (부분 입력 허용)
- **POM 없음**: POM 선정은 이 페이지에서 다루지 않음 (별도 페이지에서 관리 예정). 이 페이지는 승/패/무 raw 데이터만 등록하며, 과거에 기록된 POM 플래그가 있다면 건드리지 않고 그대로 보존함
- 등록하면 `playerStats`(개인 누적 승/패/무)와 `headToHead`(매치업 쌍별 상대전적)가 함께 갱신되고, `result.html`의 해당 년월 화면으로 바로 이동한다
- 이미 채점된 결과는 읽기 전용으로 보여주고 "다시 채점하기"로 재오픈 가능 (재채점 시 이전 기여분을 먼저 되돌린 뒤 새로 반영)
- **이 달 데이터 삭제**: 점수 등록 버튼 옆의 삭제 버튼(확인창 포함)으로 선택된 년월의 드래프트를 통째로 지울 수 있음. 이미 채점된 상태였다면 `playerStats`/`headToHead`에 반영된 기여분을 먼저 되돌린 뒤 `drafts`에서 항목을 제거한다 (되돌릴 수 없는 작업)

### `result.html` — 결과 확인 페이지

- **년월 선택**: admin.html과 동일한 드롭다운으로 년월을 골라 조회하며, 기본값은 가장 최근에 등록된 년월이다. `admin.html`에서 등록 직후 넘어올 때는 `?ym=년월` 쿼리로 방금 등록한 달을 최초 1회 우선 선택한다
- **팀 스코어보드**: 팀별로 "누적 점수"(그 팀 소속 선수들의 라운드 점수 합계)와 "승점"(선봉전~대장전 중 그 팀 소속 선수가 이긴 라운드 수, 1승당 1점)을 함께 보여준다
- **라운드별 결과 요약**: 라운드별로 "고재형 3 : 1 유다현" 한 줄씩 간단한 표로 나열 (이긴 쪽은 강조, 무승부는 별도 색)
- 비밀번호 없이 누구나 조회만 가능한 페이지이며 점수 수정 기능은 없다
- **최종 결과 확인 (POM 추첨)**: 아직 확정되지 않은 달이면 버튼을 눌렀을 때 모달이 뜨고, "누적 점수"/"승점"이 세로 릴에 담겨 위에서 아래로 빠르게 흘러간다(하스스톤 매칭 슬롯 연출 참고). 어느 기준이 뽑힐지는 모달을 열 때 `Math.random()`으로 **먼저 정해지고**, 연출은 그 결과로 수렴하도록 짜여 있다 — 빠른 스핀(5초, 모션 블러) → 멈칫(0.4초) → 슬로우모션(3초) → 두 칸 경계에서 왕복 진동(teeter, 약 1.8초)으로 **총 10초** 남짓 걸린다. 결착 패턴은 두 가지로, 승자 근처에서 그대로 확정되거나(confirm) 막판까지 지는 쪽에 걸렸다가 근소하게 뒤집힌다(flip). `prefers-reduced-motion: reduce`면 연출을 건너뛰고 결과만 바로 보여준다. 확정되면 양쪽에서 폭죽 효과 + `FFXIV_Level_Up.mp3` 효과음(50% 볼륨)과 함께, 그 기준으로 가장 높은 팀과 팀 명단이 뜨고, 그 팀에서 개인 누적 점수가 가장 높은 사람이 POM으로 화려하게 강조된다. 이 결과는 `drafts`의 `finalResult`와 `history`에 저장되며(`playerStats`의 POM 카운트도 함께 증가), 한 번 확정된 달은 다시 열어도 재추첨·재연출 없이 저장된 결과만 보여준다. 여러 사람이 동시에 열어도 먼저 저장된 결과가 이기고 나머지는 그 결과를 그대로 따른다
  - `FFXIV_Level_Up.mp3`는 `charge.mp3`/`FFXIV_Confirm.mp3`와 같은 레포 루트에 있다 (원본은 `asset/FFXIV_Level_Up.mp3`)

## 데이터 모델 — Firebase Realtime Database

최상위에 `/gachaState`(전적·드래프트), `/members`(회원 메타), `/memberPhotos`(업로드 사진 본문) 세 노드가 형제로 놓인다. 회원 정보를 `/gachaState` 밖에 둔 이유는 두 가지다.

- `reset-firebase-seed.ps1`이 `/gachaState`를 통째로 PUT해서 시드로 되돌리는데, 회원 명단까지 같이 날아가면 안 된다
- `gacha.html`은 드래프트를 공유할 때 `/gachaState` 전체를 PUT한다. 여기에 사진 data URL이 섞여 있으면 매번 수 MB를 다시 올리게 된다

사진 본문을 `/members`에서 또 한 번 분리한 것도 같은 이유로, 사진이 필요 없는 페이지(대시보드·점수 등록)가 이미지 수십 KB를 받지 않게 하기 위해서다.

### `/members`, `/memberPhotos`

```json
{
  "members": {
    "고재형": { "name": "고재형", "photo": "players/gojaehyung.png", "active": true, "order": 0, "createdAt": 1730000000000, "updatedAt": 1730000000000 },
    "새회원": { "name": "새회원", "photo": "@stored", "active": true, "order": 11, "createdAt": 1730000000000, "updatedAt": 1730000000000 }
  },
  "memberPhotos": {
    "새회원": "data:image/jpeg;base64,..."
  }
}
```

- 키는 회원 이름이며 `playerStats`/`headToHead`와 같은 이름 체계를 쓴다. RTDB 키 제약 때문에 이름에 `. $ # [ ] /` 는 넣을 수 없다 (등록 시 검사함)
- `photo`: 레포 안 파일 경로(예: `players/gojaehyung.png`)이거나, `@stored`(=실제 이미지가 `/memberPhotos/<이름>`에 있다는 표시)이거나, 빈 문자열(사진 없음)
- `active`: `false`면 "숨김" — 기록은 유지되지만 `gacha.html`의 기본 참가자 명단에서 빠진다
- `order`: 드래프트 명단에 뿌릴 순서. 없으면 이름 가나다순으로 정렬한다

### `/gachaState`

```json
{
  "playerStats": {
    "고재형": { "win": 12, "loss": 6, "draw": 0, "pom": 2 }
  },
  "headToHead": {
    "고재형|지한규": { "a": "고재형", "b": "지한규", "aWin": 2, "bWin": 0, "draw": 0 }
  },
  "drafts": {
    "2026년 7월": {
      "runId": "d...",
      "createdAt": 1730000000000,
      "yearMonth": "2026년 7월",
      "scored": false,
      "teams": [{ "name": "T1", "color": "#0a5cd8", "colorName": "딥 블루", "members": ["..."] }],
      "pairs": [["이름A", "이름B"], ["이름C", "이름D"]],
      "results": null
    }
  },
  "latestDraft": { "...": "drafts의 가장 최근 항목과 동일한 값 (구버전 호환용, 아래 설명 참고)" },
  "history": {
    "2026년 7월": {
      "runId": "d...", "yearMonth": "2026년 7월", "createdAt": 1730000000000,
      "teams": ["..."], "pairs": ["..."], "results": ["..."],
      "finalResult": { "criteria": "totalScore", "teamName": "T1", "pomName": "이름A", "decidedAt": 1730000500000 },
      "archivedAt": 1730000500000
    }
  }
}
```

- `playerStats`: 참가자별 누적 전적. 최초 시드는 `asset/DATABASE.xlsx`(과거 A팀/B팀 대결 기록, POM은 쉼표로 구분된 이름 목록)를 파싱해서 만들었다. 이 시드 데이터는 `drafts`에 대응하는 항목이 없어(=날짜 정보가 없어) admin.html의 년월 드롭다운에는 나타나지 않는다.
- `headToHead`: 두 사람이 실제로 맞붙어서 나온 승/패/무 누적. 키는 두 이름을 `|`로 이은 것이며 순서는 상관없이 양방향으로 조회 가능하게 설계되어 있다.
- `drafts`: 년월(`yearMonth`)을 키로 하는 드래프트 히스토리. `gacha.html`에서 새 드래프트를 돌릴 때마다 해당 년월 키에 기록되며, admin.html은 이 맵 전체를 드롭다운으로 보여주고 어떤 월이든 선택해 채점/재채점할 수 있다. 같은 달에 두 번 드래프트를 돌리면 그 달의 항목은 덮어써진다 (달 단위로는 한 건만 보관). admin.html에서 "이 달 데이터 삭제"를 누르면 이 맵에서 항목이 통째로 사라진다.
  - `results`: 라운드(=`pairs`의 각 쌍)별로 두 선수의 점수를 비교해 자동으로 산출된 `{ 이름: { score, outcome } }` 형태. `outcome`은 `"win" | "loss" | "draw"`. `result.html`에서 POM이 확정되면 그 사람의 항목에 `pom: true`가 추가된다.
  - `finalResult`: `result.html`에서 "최종 결과 확인" 슬롯이 멈추고 나면 기록되는 `{ criteria, teamName, pomName, decidedAt }`. `criteria`는 `"totalScore" | "winPoints"` 중 슬롯이 뽑은 기준이고, `teamName`/`pomName`은 그 기준으로 정해진 우승팀/POM이다. 한 번 정해지면 고정이며, `result.html`은 이 필드가 있으면 다시 뽑지 않고 저장된 값을 그대로 보여준다.
- `latestDraft`: 이 기능(년월별 히스토리, 2026-07 도입) 이전에 쓰이던 필드로, `drafts`와 동일한 값을 가리키는 구버전 호환용 미러다. `gacha.html`은 새 드래프트마다 `drafts`와 `latestDraft`를 함께 갱신하고, admin.html은 `drafts`에 아직 없는 년월이 `latestDraft`에 있으면(=이 기능 도입 전에 생성된 데이터) 폴백으로 병합해 보여준다. 새로 저장되는 채점 결과는 `drafts`에만 반영된다.
- `history`: `result.html`에서 POM까지 확정된(=`finalResult`가 생긴) 드래프트만 영구 보관하는 별도 아카이브. `drafts`와 달리 admin.html의 "이 달 데이터 삭제"로도 지워지지 않는다 — 채점이 끝난 달의 최종 기록을 나중에 그 달 데이터를 정리하더라도 잃지 않기 위한 것. 년월당 한 번만 기록되며(멱등), 아직 없는 경우에만 채워진다.
- `pairs`는 각 대결이 실제로 누구와 누구였는지를 기록해, 관리자 점수 등록 시 `headToHead`를 자동으로 갱신하는 데 쓰인다.

## 공유 방식 (Firebase Realtime Database, 완전 오픈)

모든 페이지가 별도 백엔드 없이, Firebase Realtime Database REST API로 직접 읽고/쓴다.

- 베이스 URL: `https://team-draft-1ab3a-default-rtdb.firebaseio.com` (`/gachaState.json`, `/members.json`, `/memberPhotos.json`)
- 규칙: `{ ".read": true, ".write": true }` — 인증/토큰 없이 누구나 읽고 쓸 수 있다. 친구 그룹 캐주얼 트래커라 보안보다 편의성을 우선한 선택.
- 읽기: `fetch(FIREBASE_URL + "/gachaState.json")`, 쓰기: 같은 경로에 `PUT` (전체 객체를 통째로 덮어씀).
- 회원 관련 쓰기는 하위 경로 단위로 한다: `PUT /members/<이름>.json`, `PUT /memberPhotos/<이름>.json`, `DELETE /members/<이름>.json`. `admin-stats.html`의 전적 저장은 바뀐 사람만 담아 `PATCH /gachaState/playerStats.json` 한다.
- 예전에는 이 레포의 `gacha-state.json`을 GitHub Contents API로 읽고/쓰며 GitHub 토큰이 필요했으나, 지금은 Firebase로 완전히 이전해 토큰 개념이 없다. (핀볼 페이지 `index.html`/`state.json`은 여전히 예전 GitHub 방식을 그대로 쓴다.)
- 레포에 남아있는 `gacha-state.json`은 더 이상 앱이 읽지 않는 **과거 시드 데이터 스냅샷**일 뿐이다 (참고용).

## 알려진 제한 사항

- `drafts`는 같은 달에 새 드래프트를 다시 돌리면 그 달 항목이 덮어써진다 (달 단위 히스토리이지, 달 안에서 여러 건을 보관하진 않는다).
- **완전 오픈 데이터스토어이므로 URL을 아는 사람은 누구나 데이터를 읽고 쓸 수 있다.** 토큰 없이 편하게 쓰는 대신 감수한 트레이드오프. 관리자 페이지의 비밀번호(`daft`)는 클라이언트 측 실수 방지 게이트일 뿐, Firebase 규칙 자체를 막지는 못한다 — 비밀번호를 몰라도 REST API로 직접 쓰면 우회 가능하다. 회원 사진도 마찬가지로 URL만 알면 누구나 받아볼 수 있다.
- 회원 이름 변경은 `/gachaState` 전체를 읽어 치환한 뒤 통째로 다시 쓰기 때문에, 바로 그 순간 다른 사람이 채점을 저장하고 있으면 한쪽이 덮어써질 수 있다. 사람이 몰릴 일이 없는 규모라 낙관적 갱신으로 두었다.
- 업로드 사진은 320×320 JPEG data URL로 RTDB에 들어간다. 수십 명 규모까지는 문제없지만, 회원이 크게 늘면 `/memberPhotos`를 통째로 받는 방식(회원 관리·회원 데이터 페이지) 대신 필요한 것만 받도록 바꿔야 한다. `gacha.html`은 이미 업로드 사진을 쓰는 회원이 있을 때만 이 노드를 받아온다.
- `admin-stats.html`에서 고친 누적 전적은 `drafts[*].results`의 라운드별 기록과 어긋날 수 있다 (누적값을 직접 덮어쓰는 것이라 역산·검증을 하지 않는다).
- 이 README가 다루는 기능은 로컬 정적 서버 또는 GitHub Pages 배포본에서 검증되었다. 로컬 파일 더블클릭(`file://`)으로는 Firebase fetch가 막혀 전적 연동이 동작하지 않는다.
